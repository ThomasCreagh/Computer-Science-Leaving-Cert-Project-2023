Meeting the Brief

Basic requirements

  1. I have designed and developed an interactive computer model based on the game Connect Four. The user can interact with my model through the console so that they can play the game to completion. The images below show three examples of the end states of my game in which a human player plays against a computer player. I imported the library 'random' (shown in the image to the right) in order to achieve randomness in my model allowing it to behave differently each time it is run.

  2. My model accepts more than three different inputs. These inputs can be but are not limited to, (i) the dimensions of the game, (ii) what mode the game is played in and, (iii) what column the user would like to slot their piece into. The inputted data is validated using functions to check whether the inputs are in the correct data type. They also check if the inputs are in the appropriately defined validation rules. An example of this can be seen when the user is inputting the number of rows they would like to play with, it checks the data to make sure it is an integer and if it is under the ten thousand limit.

  3. My game is interactive and operable in a variety of modes, single-player, multi-player and simulation-play. If 'simulation-play' is chosen, the user is prompted to input the number of games the simulation will iterate through. The addition of 'simulation-play' allows data to be efficiently collected across many games.

Advanced requirements

  1. The next requirement I set out to complete was to extend the basic model so that each time it is run, it stores the resulting data from the game in a CSV file. This CSV file contains the number of moves it took to complete a game and how the player won the game i.e. horizontally, vertically, diagonally or if it was a draw.

  2. I developed algorithms that perform some statistical analysis on the data I collected from each game. I found the mean, the median, the mode, and the standard deviation of the number of moves it takes to win the game. That data is displayed through the console but also displayed through a 'matplotlib' histogram in a graphical format, which displays the number of moves per game.

  3. My initial hypothesis was, the player puts their first piece into the middle column, they would have a higher success rate of winning than a player who puts their first piece on the side columns of the board. I used simulation-play mode to test my hypothesis by collecting data on 10,000 games. This data allowed me to prove whether my hypothesis was correct or not. I predicted if I changed the parameters, by increasing the board size, it wouldn't affect the results of the hypothesis. The results are presented in my report.

Investigation and Plan

The thematic brief requires you to develop an interactive model based on games. The brief gave different examples, such as cards, dice, board, or children's games.

References [1] - [5] considered the concept of a physical model and simulation. Researching the video about Simulating Natural Selection [4] helped me understand simulation is based on defined rules which use randomness. It led me to believe I could implement Natural Selection via generational neural networks. On further research of the brief, I realised that including generational neural networks wasn't necessary. Investigating analytics [6] informed me of the use of data analytics to predict a simulated model's outcome. From [7] I learned that game design methodology is based on the level of abstraction of game mechanics. I also read some python websites on how to use classes to develop a game [8], [9]. During this project, I used my course book for data analytics [10]. In addition, I have used w3schools [11] to help me with creating my website navbar.

After reviewing the brief, I developed some preliminary selection criteria:

  1. The difficulty and feasibility of designing and implementing a model of the game.
  2. How I was going to abstract the rules of the game.
  3. Could I simplify the game if it was too complex.
  4. The suitability of the game data I would collect for modelling, simulation and analysis.

I carried out several web searches for the best game that would fit the points above. I considered the following games:

  1. Rock, Paper, Scissors.
  2. Chess.
  3. Connect Four.

I found game 1, Rock, Paper, Scissors, to be too simple and that there was no strategy involved. It would be difficult for me to generate usable data to formulate a hypothesis. When evaluating game 2, Chess, I realised that it wouldn't be feasible to implement the game as chess is very strategic and too complicated. On reviewing game 3, Connect Four, I determined it would fit all my preliminary selection criteria.

My objective for my project was to implement Connect Four as a simple text-based interactive interface. I would test my hypothesis by analysing data stored in a CSV file which includes game outcome data.

I've created a lot of games in the past and my concern was that Connect Four would be too simple using the original board size. Therefore I determined that I should make my game more difficult. To achieve this the board size could be increased at the choice of the player, and the number of connecting pieces to win could be increased. That is why I called my game 'Connect X'.

Design

I first created a high-level flowchart chart to visualise the structure of my game. This allowed me to model the essential parts and implement the main code loop. It helped me identify the likely parts of my code I would integrate into individual functions. My flowchart is shown in the diagram below. It captures the operations of the multi-player mode.


High level flowchart

The only difference between the multi-player and the other two modes is that, instead of asking the human player for the column choice, the other modes would pick a random available column to slot its piece into when it's the computer's turn.

If a human player tries to input a piece into an invalid column, the program prompts the user to select a column from a valid list of options.

I designed my program to only iterate once when it is the computer players turn to pick a random input. It does this by gathering the list of columns available to be slotted into and picking a random index from that list; by implementing it like this, it is more efficient than iterating through a 'hard coded' list of numbers.

When modelling 'Connect X', I found that the most important design decision I had to make was how my program displayed the board. I realised that printing the board to the console was all that was necessary to meet my objective of a simple text-based interface. Creating a graphical display wasn't a requirement of the brief.abstraction described in the brief.

I designed the game to fit the original board size at first and then adapted it to work for any board size and any number of connecting pieces to win. I also cleaned up my code by implementing classes. The first part of the game 'start.py' sets up the game so that when 'game.py' gets called to run, it has all the parameters the user has chosen. The game file automatically temporarily stores information about each game being run and appends the data to a CSV file for future data analytics. I use the libraries matplotlib and seaborn to create my graphs.

Implementation

I used an iterative approach to develop my artefact. After I abstracted rules, I decomposed my implementation and wrote down these six clearly defined steps that enabled me to implement my project objectives in a modular fashion:

  • Replicating Connect Four through my abstracted rules with multi-player.
  • Adding single-player and simulation-play.
  • Enable the user to change the size of the board.
  • Enable the user to change the number of connecting pieces to win.
  • Including Data capture.
  • Including Data analysis and visualisation.

As I was developing my game, I hit two difficult problems. Firstly I had problems with repeating code. Adding functions helped separate the code into blocks. The second problem was, I had to make lots of variables global in each function, or else I had to import all the necessary parameters into each function. This made it hard to develop the game further as it was cluttered. Classes solved my problems because I had class variables that would work for every function if I imported 'self' into their parameters.

The most complex and important algorithm I implemented was the function to check if the game was finished. This was particularly difficult in Connect Four because it was an exhaustive algorithm: vertically, horizontally and diagonally. Enabling the user to change the size of the board and to change the number of connecting pieces to win made the logic more difficult. Using abstraction, I represented my board using a list of lists also known as a matrix or a 2D array. This made the game easier to implement.

When programming Connect Four with the original dimensions, I had an if statement to check for four connecting pieces. This code wouldn't work if you increased it to Connect Five. I solved this by adding a for loop which looped over the if statement.

The for loops in the function know to iterate through all the moves via the following algorithms in the file 'game.py':

  1. For each row check the number of possible horizontal wins: rows * (row length - connected pieces to win) = the total number of horizontal wins possible (lines 119 - 128)
  2. For each column check the number of possible horizontal wins: columns * (column_length - connected pieces to win) = the total number of vertical wins possible (lines 130 - 140)
  3. For each row check the number of possible horizontal wins: 2 * ((row length - (connected pieces to win- 1)) * (column length - (connected pieces to win - 1))) = the total number of diagonal wins possible (lines 142 - 168)
  4. If none of those is true and the board is filled, it considers the game to be a draw. (lines 170 - 174)

My next step was to develop the code for data analytics. I developed my analytics in a way that would test my hypothesis. I added code to append game data into a CSV file, such as: the end state of the game, what player won the game, and what random column they selected first when in simulation-play mode. After running 10,000 games, I tallied up the total wins for each column and divided that by the number of games (10,000). The results are presented in the evaluation section.

I also got a few of my friends to try and break my game in any way through the interface. They acted as the game's alpha testers.

Unit Test Results

The original game used colour to help the user distinguish the difference between the pieces. In the console, I only had the use of one colour. I originally labelled the pieces with 'R' being red and 'Y' being yellow. When my friends were testing my game they found it difficult to see the difference between the 'R' and the 'Y'. The game X's & O's has a similar problem: it has no colours but has two very distinct shapes that are easy to spot among each other. Once I realised how effective it was, it led me to change my design and use an 'X' and an 'O' as my pieces as a result of my testing.

Evaluation

While evaluating my project I looked back at the brief. I have created a model to be an exact replica of the real-world physical game Connect Four. I investigated different games and after picking Connect Four I designed a model, abstracted the game and implemented the game in python with analytics. I increased the difficulty of the game to meet more of what I thought the brief wanted.

I ran the game 10,000 times in simulation-play mode and stored all the data in a CSV file. Using matplotlib I created a line distribution to display the results to check if my hypothesis was correct. Clearly, it wasn't and was practically the opposite of my hypothesis.

Looking at the graphs, placing your first piece in the middle had the worst probability of winning. There is an advantage if you place your first piece on the sides of the board. This proved my hypothesis wrong.

I also predicted that if I changed the parameters, by increasing the board size, it wouldn't affect the results of the hypothesis. This was proven to be right.

I think my project could be improved in various ways. One of the biggest factors is the difficulty of the game. My current game only uses random inputs as its strategy. This makes it really easy to win the game against the computer because the computer is very naive. In order to overcome this, I would either have to hard code an exhaustive, rule-based strategy or create some sort of machine learning to think ahead and act as a more difficult opponent. If I implemented these changes it would make it act more like a real-world simulation model.

Other small features I could add to improve my game are:

  • Add hints if wanted.
  • Develop a graphical interface with universal design.
  • Be able to undo a move if you miss-clicked.
  • Include a leaderboard system to check the best records.
